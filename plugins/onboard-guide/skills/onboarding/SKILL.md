# onboarding — 레거시 코드 온보딩 문서 자동 생성

현재 작업 중인 레포지토리(cwd)를 분석하여, 신규 투입자가 코드 구조를 빠르게 이해하고 **직접 실행해 볼 수 있도록**
돕는 **단일 파일 인터랙티브 HTML 문서**를 레포 루트에 `onboarding.html`로 생성하라.

추가 인자: $ARGUMENTS (특정 디렉토리나 관점 지정 시 해당 부분 집중 분석)

## 0단계: 분석 범위 설정 (먼저 수행)

분석을 시작하기 전에 범위를 정하라. 무작정 전체를 읽지 말 것.

1. **제외 대상**: 다음은 트리 표기만 하고 내용 분석에서 제외한다 —
   `node_modules`, `.git`, `dist`, `build`, `out`, `.next`, `target`, `.venv`/`venv`, `__pycache__`,
   `vendor`, `coverage`, `.cache`, 빌드 산출물, 미니파이 파일, 바이너리/이미지/폰트.
   락파일(`package-lock.json`, `poetry.lock`, `Cargo.lock` 등)은 **의존성 데이터로만 참고**하고 정독하지 않는다.
2. **규모 측정**: `git ls-files | wc -l` (또는 `find` + 제외)로 소스 파일 수를 센다.
   - **~300 파일 이하**: 전체 분석
   - **300~1500 파일**: 디렉토리 크기·import 빈도 상위 위주로 샘플링, 트리는 2 depth로 제한
   - **1500 파일 초과**: 진입점과 상위 의존 모듈에만 집중하고, 문서에 "부분 분석" 배지를 표기
3. **모노레포 감지**: `pnpm-workspace.yaml`, `lerna.json`, `turbo.json`, `go.work`, Cargo workspace, `packages/*`, `apps/*` 등이 있으면 모노레포로 보고 —
   최상위에서 **패키지 목록·역할·상호 의존**을 먼저 개관한 뒤, $ARGUMENTS로 지정된(없으면 가장 핵심적인) 패키지 1개를 깊게 분석한다.
4. **집중 범위**: $ARGUMENTS가 주어지면 해당 경로/관점에 집중하고, 나머지는 개요 수준으로만 다룬다.

## 1단계: 레포 분석

다음 순서로 코드베이스를 파악하라. **추측하지 말고 실제 파일을 근거로 작성할 것.**

1. **메타데이터**: 프로젝트 목적·의존성·진입점을 매니페스트에서 파악. 언어별로 폭넓게 탐지 —
   `README*`, `package.json`, `pyproject.toml`/`setup.py`/`requirements*.txt`, `go.mod`, `Cargo.toml`,
   `composer.json`, `Gemfile`, `pom.xml`/`build.gradle(.kts)`, `*.csproj`/`*.sln`, `CMakeLists.txt`,
   `Makefile`, `Dockerfile`, `docker-compose*.yml`.
2. **실행·개발 환경 파악** (필수): 신규자가 바로 돌려볼 수 있도록 다음을 실제 파일에서 추출 —
   - **사전 요구사항**: 런타임/버전(`.nvmrc`, `.python-version`, `go` 버전, `engines` 등)
   - **설치**: 의존성 설치 명령 (`npm install`, `uv sync`, `go mod download` 등)
   - **실행/빌드**: 앱 실행·빌드 명령 (`package.json`의 `scripts`, `Makefile` 타깃, `docker-compose` 등에서 확인된 것만)
   - **테스트**: 테스트 실행 명령과 테스트 위치
   - **환경 변수**: `.env.example`/`.env.sample`/README에서 필요한 키 목록 (값은 노출 금지, 키 이름과 용도만)
3. **디렉토리 구조**: 2~3 depth 트리. 각 디렉토리의 **역할을 한 줄로** 정리 (제외 대상은 접어서 간단히 표기)
4. **핵심 파이프라인**: 대표 실행 흐름 1~2개 추적
   - 진입점 → 핵심 처리 → 외부 I/O → 결과 반환 순으로 5~8단계
   - 각 단계마다: 담당 파일, 핵심 함수/클래스(실제 시그니처), 코드 읽을 때 주목할 포인트
5. **핵심 모듈**: 상위 5~9개 파일 선정. 선정 기준은 **① 코드량 ② import/의존 빈도 ③ 변경 빈도(churn)** 를 함께 사용.
   churn은 가능하면 `git log --pretty=format: --name-only | sort | uniq -c | sort -rg | head` 로 참고 (git 없으면 생략).
   각 모듈의 역할, 주요 클래스/함수, 모듈 간 의존 관계 정리.
6. **읽기 순서**: "여기부터 읽으세요" — 신규 투입자가 읽어야 할 파일 순서와 이유
7. **엣지케이스**: 진입점이 불명확하거나 소스가 거의 없으면, 추측으로 채우지 말고 문서에 "진입점 불명확 — 확인 필요" 로 솔직히 표기한다.

## 2단계: HTML 생성

분석 결과를 레포 루트의 `onboarding.html` **단일 파일**로 생성하라. 외부 CDN 없이 vanilla HTML/CSS/JS만 사용 (사내망에서도 열리도록).
`onboarding.html`이 이미 있으면 덮어쓰되, 완료 보고에 "기존 파일 덮어씀"을 명시한다.

### 필수 UI 구성

- **레이아웃**: 좌측 사이드바(디렉토리 트리) + 우측 메인 패널. 다크 톤 상단바 + 핵심 지표(KPI) 스트립(파일 수, 코드 라인, 주요 언어, 계층 수, 의존성 수, 예상 소요). 부분 분석인 경우 "부분 분석" 배지 표기
- **시작하기 (Getting Started)**: 문서 상단 눈에 띄는 위치에 배치. 1단계에서 추출한 **사전 요구사항 → 설치 → 실행/빌드 → 테스트** 명령을 복사 가능한 코드 블록으로, 필요한 **환경 변수 키 목록**(용도 포함, 값은 제외)을 표로 제공. 신규자가 문서만 보고 프로젝트를 돌려볼 수 있어야 한다
- **프로젝트 개요 (상세)**: 단순 한 줄 소개가 아니라 4가지를 카드로 — ① 전체 아키텍처/계층 구조와 설계 철학, ② 핵심 데이터 흐름(어떤 객체가 어떻게 변환되며 흐르는지), ③ 공식 확장 지점(커스터마이징은 어디서), ④ 외부 핵심 의존성과 각각의 역할
- **전체 구조 워크플로우 그래프 (SVG)**: 의존 관계를 계층(사용자 → API → 코어 → 전송 → 외부 라이브러리 등)으로 배치. 실선=호출 흐름, 점선=참조. **노드 클릭 시 해당 모듈 카드로 스크롤·하이라이트** 연동. 범례 포함
- **디렉토리 트리 (사이드바)**: 클릭으로 접고 펼치기. 각 항목에 역할 설명. 핵심 파일은 클릭 시 해당 모듈 상세로 스크롤/하이라이트
- **파이프라인 다이어그램 (메인)**: 실행 흐름을 단계별 노드로. **노드 클릭 시 상세 패널**(파일 경로, 핵심 함수, 읽기 포인트, 다음 단계 호출 코드)이 펼쳐짐
- **모듈 카드**: 핵심 모듈별 카드. 클릭 시 상세 토글. 각 카드에 반드시 포함:
  - **⭐ 최우선으로 볼 함수/클래스 1개** + 왜 그것부터 봐야 하는지 이유
  - **실제 코드 스니펫**: 핵심 함수의 실제 코드 5~8줄을 구문 강조하여 인용하고, `# 주석`으로 각 줄 설명 (코드는 반드시 원본 발췌, 창작 금지)
  - **정독/스킵 가이드**: 반드시 읽을 부분과 필요할 때만 볼 부분을 태그로 구분
  - **난이도 표시**(●●● 3단계)와 **코드량(loc)**
  - 그 외 주요 함수/클래스 (실제 행 번호 포함)
  - **의존성 구분 표기**: 내부 모듈 의존(파란색) vs 외부 라이브러리 의존(주황색). 외부는 어떤 라이브러리의 어떤 기능을 쓰는지까지 명시
- **읽기 순서 가이드**: 번호 매겨진 학습 경로. 각 단계에 예상 소요와 "왜 이 순서인지" 이유
- **검색**: 입력창에 키워드 입력 시 트리/모듈 카드 필터링

### 정확성 (필수)

- **HTML 이스케이프**: 코드 스니펫·파일 경로·식별자를 HTML에 넣을 때 `<`, `>`, `&`, `"`, `'` 를 반드시 엔티티로 이스케이프한다. 이스케이프를 빠뜨리면 코드가 태그로 해석되어 렌더가 깨진다
- **환각 금지**: 모든 파일 경로, 함수명, 행 번호, 실행 명령은 실제 코드/설정에서 확인된 것만 사용
- 다이어그램은 SVG 또는 CSS로 직접 그리기 (이미지 파일·외부 요청 의존 금지)

### 디자인 기준 (엔터프라이즈)

- 절제된 팔레트: 뉴트럴(slate) 계열 + 브랜드 색 1개 + 외부의존 강조색 1개. 그 이상 색을 늘리지 말 것
- flat 디자인: 그라디언트·질감·과한 그림자 금지. 카드 경계선과 미세한 그림자로만 계층 표현
- 코드/식별자는 모노스페이스, 본문은 시스템 산세리프. 타이포 스케일 일관성 유지
- **반응형**: 데스크톱/태블릿/모바일에서 요소 겹침·글자 넘침·가로 스크롤이 없도록 미디어 쿼리 처리
- **접근성(a11y)**: 시맨틱 태그(`nav`/`main`/`section`) 사용, 접고 펼치는 요소는 키보드로 조작 가능(`button`/`tabindex`+`aria-expanded`), 텍스트 대비 WCAG AA 충족, 검색 입력에 `label`

### 품질 기준

- 설명은 한국어, 코드 식별자는 원문 유지
- 신규 투입자 관점: "이 코드가 뭘 하는지"보다 "**어디부터 어떻게 읽고, 어떻게 실행하는지**"에 초점

## 3단계: 검증

생성 후 다음을 자가 점검하고 결과를 보고하라:

- [ ] HTML에 등장하는 파일 경로가 모두 실제 존재하는가 (`ls`로 확인)
- [ ] 파이프라인 각 단계의 함수가 해당 파일에 실제 존재하는가 (`grep`으로 확인)
- [ ] "시작하기"의 설치/실행/테스트 명령이 실제 설정(`package.json` scripts, `Makefile` 등)과 일치하는가
- [ ] 인용한 코드 스니펫이 원본과 일치하고, HTML 특수문자가 이스케이프되어 렌더가 깨지지 않는가
- [ ] 외부 요청이 0인가 (CDN·웹폰트·원격 이미지·fetch 없음 — 사내망 오프라인에서 열림)
- [ ] `.env` 값 등 민감정보가 문서에 노출되지 않았는가 (키 이름/용도만)
- [ ] id 중복이 없고, 클릭 연동(트리/그래프 → 카드, 파이프라인 노드)이 모두 매칭되는가
- [ ] 접고 펼치는 요소가 키보드로 조작되는가
- [ ] 데스크톱(1440px)·태블릿(900px)·모바일(390px) 폭에서 요소 겹침·글자 넘침·가로 스크롤·텍스트 잘림이 없는가

완료 후 생성된 `onboarding.html`의 경로를 보고하고, 기존 파일을 덮어썼다면 그 사실을 함께 알린다.
