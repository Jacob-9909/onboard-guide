# onboarding — 레거시 코드 온보딩 문서 자동 생성

현재 작업 중인 레포지토리(cwd)를 분석하여, 신규 투입자가 코드 구조를 빠르게 이해할 수 있는
**단일 파일 인터랙티브 HTML 문서**를 레포 루트에 `onboarding.html`로 생성하라.

추가 인자: $ARGUMENTS (특정 디렉토리나 관점 지정 시 해당 부분 집중 분석)

## 1단계: 레포 분석

다음 순서로 코드베이스를 파악하라. **추측하지 말고 실제 파일을 근거로 작성할 것.**

1. **메타데이터**: README, pyproject.toml/package.json/build.gradle/go.mod 등에서 프로젝트 목적, 의존성, 진입점(entry point) 파악
2. **디렉토리 구조**: 2~3 depth까지 트리를 만들고, 각 디렉토리의 **역할을 한 줄로** 정리 (테스트/문서/빌드 산출물 등은 간단히 표기)
3. **핵심 파이프라인**: 대표적인 실행 흐름 1~2개를 추적
   - 진입점 → 핵심 처리 → 외부 I/O → 결과 반환 순으로 5~8단계
   - 각 단계마다: 담당 파일, 핵심 함수/클래스(실제 시그니처), 코드 읽을 때 주목할 포인트
4. **핵심 모듈**: 코드량/의존도 기준 상위 5~9개 파일에 대해 역할, 주요 클래스/함수, 모듈 간 의존 관계 정리
5. **읽기 순서**: "여기부터 읽으세요" — 신규 투입자가 읽어야 할 파일 순서와 이유

## 2단계: HTML 생성

분석 결과를 레포 루트의 `onboarding.html` **단일 파일**로 생성하라. 외부 CDN 없이 vanilla HTML/CSS/JS만 사용 (사내망에서도 열리도록).

### 필수 UI 구성

- **레이아웃**: 좌측 사이드바(디렉토리 트리) + 우측 메인 패널. 다크 톤 상단바 + 핵심 지표(KPI) 스트립(파일 수, 코드 라인, 주요 언어, 계층 수, 의존성 수, 예상 소요)
- **프로젝트 개요 (상세)**: 단순 한 줄 소개가 아니라 4가지를 카드로 정리 — ① 전체 아키텍처/계층 구조와 설계 철학, ② 핵심 데이터 흐름(어떤 객체가 어떻게 변환되며 흐르는지), ③ 공식 확장 지점(커스터마이징은 어디서 하는지), ④ 외부 핵심 의존성과 각각의 역할
- **전체 구조 워크플로우 그래프 (SVG)**: 코드 구조 기반 의존 관계를 계층(사용자 → API → 코어 → 전송 → 외부 라이브러리 등)으로 배치. 실선=호출 흐름, 점선=참조. **노드 클릭 시 해당 모듈 카드로 스크롤·하이라이트** 연동. 범례 포함
- **디렉토리 트리 (사이드바)**: 클릭으로 접고 펼치기. 각 항목에 역할 설명. 핵심 파일은 클릭 시 해당 모듈 상세로 스크롤/하이라이트
- **파이프라인 다이어그램 (메인)**: 실행 흐름을 단계별 노드로 시각화. **노드 클릭 시 상세 패널**(파일 경로, 핵심 함수, 읽기 포인트, 다음 단계 호출 코드)이 펼쳐짐
- **모듈 카드**: 핵심 모듈별 카드. 클릭 시 상세 토글. 각 카드에 반드시 포함:
  - **⭐ 최우선으로 볼 함수/클래스 1개** + 왜 그것부터 봐야 하는지 이유
  - **실제 코드 스니펫**: 핵심 함수의 실제 코드 5~8줄을 구문 강조하여 인용하고, `# 주석`으로 각 줄 설명 (코드는 반드시 원본 발췌, 창작 금지)
  - **정독/스킵 가이드**: 반드시 읽을 부분과 필요할 때만 볼 부분을 태그로 구분
  - **난이도 표시**(●●● 3단계)와 **코드량(loc)**
  - 그 외 주요 함수/클래스 (실제 행 번호 포함)
  - **의존성 구분 표기**: 내부 모듈 의존(파란색) vs 외부 라이브러리 의존(주황색). 외부는 어떤 라이브러리의 어떤 기능을 쓰는지까지 명시
- **읽기 순서 가이드**: 번호 매겨진 학습 경로. 각 단계에 예상 소요와 "왜 이 순서인지" 이유
- **검색**: 입력창에 키워드 입력 시 트리/모듈 카드 필터링

### 디자인 기준 (엔터프라이즈)

- 절제된 팔레트: 뉴트럴(slate) 계열 + 브랜드 색 1개 + 외부의존 강조색 1개. 그 이상 색을 늘리지 말 것
- flat 디자인: 그라디언트·질감·과한 그림자 금지. 카드 경계선과 미세한 그림자로만 계층 표현
- 코드/식별자는 모노스페이스, 본문은 시스템 산세리프. 타이포 스케일 일관성 유지
- **반응형**: 데스크톱/태블릿/모바일에서 요소 겹침·글자 넘침·가로 스크롤이 없도록 미디어 쿼리 처리

### 품질 기준

- 모든 파일 경로, 함수명, 행 번호는 실제 코드에서 확인된 것만 사용 (환각 금지)
- 다이어그램은 SVG 또는 CSS로 직접 그리기 (이미지 파일 의존 금지)
- 설명은 한국어, 코드 식별자는 원문 유지
- 신규 투입자 관점: "이 코드가 뭘 하는지"보다 "**어디부터 어떻게 읽어야 하는지**"에 초점

## 3단계: 검증

생성 후 다음을 자가 점검하고 결과를 보고하라:

- [ ] HTML에 등장하는 파일 경로가 모두 실제 존재하는가 (`ls`로 확인)
- [ ] 파이프라인 각 단계의 함수가 해당 파일에 실제 존재하는가 (`grep`으로 확인)
- [ ] 인용한 코드 스니펫이 원본과 일치하는가 (창작·환각 금지)
- [ ] id 중복이 없고, 클릭 연동(트리/그래프 → 카드, 파이프라인 노드)이 모두 매칭되는가
- [ ] 데스크톱(1440px)·태블릿(900px)·모바일(390px) 폭에서 요소 겹침·글자 넘침·가로 스크롤·텍스트 잘림이 없는지 점검
